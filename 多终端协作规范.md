# AgentShield 多终端协作规范

> 多 Claude 终端并行开发时的行为约束、代码规范与协作协议。

---

## 一、Git 工作流

### 1.1 核心规则

```
一次只做一个模块。
一个终端 = 一个 worktree = 一个分支 = 一个模块。
绝不跨模块修改文件。
```

### 1.2 分支命名

```
feat/<module-slug>

示例:
  feat/checkpoint-snapshot    # 断点续跑快照机制
  feat/token-quota            # Token 算力配额管理
  feat/selinux-mac            # SELinux MAC 完整策略
  feat/faiss-vector           # FAISS 向量检索
  feat/iam-abac               # 企业 IAM + ABAC JIT
```

### 1.3 工作流程

```bash
# --- 开发期 ---
git add <specific-files>        # 只 add 本模块文件，禁止 git add -A
git commit -m "<格式见下>"      # 小步提交，每个逻辑单元一个 commit
# 重复 coding → commit 直到模块完成

# --- 完成期 ---
cargo build -p <crate>         # Rust: 整个 crate 编译通过
go build ./...                  # Go: 整个 module 编译通过
# 运行本模块测试
git push origin <branch>       # 推送到远程

# --- 通知人类 review ---
# 在终端输出: "=== MODULE COMPLETE: feat/xxx === 请 review PR"
```

### 1.4 Commit 格式

```
<type>(<scope>): <简短描述>

<详细说明（可选）>

Co-Authored-By: Claude <noreply@anthropic.com>
```

type: `feat` | `fix` | `refactor` | `test` | `chore` | `docs`
scope: 模块名，如 `checkpoint` | `token-quota` | `faiss`

示例：
```
feat(checkpoint): add CheckpointManager with journal serialization

Implements State Journal + Workspace Layering checkpoint system.
- journal.rs: MessagePack-based conversation state serialization
- manifest.rs: BLAKE3 file manifest with workspace tracking
- recovery.rs: rollback orchestration with remediation prompts

Co-Authored-By: Claude <noreply@anthropic.com>
```

### 2.3 公共文件冲突处理

以下文件可能被多个模块修改，**由人类在后整合阶段处理**：

| 公共文件 | 可能冲突的模块 | 处理方式 |
|---------|-------------|---------|
| `deployments/docker-compose.yml` | D1, D2, 所有依赖新 service 的模块 | D1/D2 各自添加自己的 service 块；人类用 `git merge` 合并 |
| `management-server/internal/api/router.go` | B1, B2, B3, B4, B5 | 各自添加不同路由组；人类手工合并 |
| `management-server/web/src/App.tsx` | B1, B3, D3 | 各自添加不同 Route；人类手工合并 |
| `management-server/go.mod` | B1, D1, D2 | `go mod tidy` 解决 |
| `ml-pipeline/src/agentshield_ml/api/main.py` | C2, C3, C4 | 各自添加不同路由组；人类手工合并 |
| `ml-pipeline/pyproject.toml` | C3 | 只有 C3 会改，无冲突 |

> **终端守则**：如果你需要修改公共文件，只在你负责的代码块内变更（如 router.go 中只添加你的路由注册行），不要重排已有代码、不要修改已有 import、不要格式化整个文件。

## 三、代码质量标准

### 3.1 通用规则

# CLAUDE.md

Behavioral guidelines to reduce common LLM coding mistakes. Merge with project-specific instructions as needed.

**Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

---

**These guidelines are working if:** fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.

## 四、测试要求

### 4.1 必须写

| 语言 | 测试位置 | 运行命令 |
|------|---------|---------|
| Rust | 同文件 `#[cfg(test)] mod tests` | `cargo test -p agent-runtime` |
| Go | 同包 `_test.go` | `go test -race ./...` |
| Python | `tests/` 目录 | `pytest` |
| TS | 暂不做测试 | — |

### 4.2 测试覆盖率要求

- Rust: 至少 coverage 新增模块的核心逻辑（buffer 操作、序列化往返、状态转换）
- Go: 至少 coverage Store 操作、配置加载、handler business logic
- Python: 至少 coverage 模型前向传播、API 输入-输出一致性

### 4.3 每个模块的测试基线

```
完成模块前，必须满足:

☐ 编译通过（Rust: cargo build / Go: go build / Python: import 无报错）
☐ 新增测试通过
☐ 现存测试未因本模块修改而失败
☐ 无 clippy/vet/ruff/tsc 告警
```
