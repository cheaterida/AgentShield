# AgentShield 并行开发计划

> 基于 `development-progress.md` 中 16 个未完成模块的依赖分析与并行度规划。

---

## 一、模块清单与归属

共 16 个模块，按组件归属拆分为四条独立开发线：

| Track | 组件栈 | 目标模块数 |
|-------|--------|-----------|
| **Track A — 内核/Rust 线** | agent-runtime (Rust) + kernel-hardening (SELinux) | 3 |
| **Track B — Go 后端线** | management-server (Go) + OPA (Rego) + 部分前端 | 5 |
| **Track C — Python ML 线** | ml-pipeline (Python, PyTorch, DGL, FAISS) | 4 |
| **Track D — 基础设施/前端线** | DevOps (Docker, DB) + React 前端 | 3 (+ 部分前端共享) |

## 四、各 Track 详细任务

### Track A — 内核/Rust 线

| 序号 | 模块 | 工时 | 前置依赖 | 关键文件 |
|------|------|------|---------|---------|

| A3 | MAC 文件安全回收站 | 5d | A2 | `kernel-hardening/selinux/` + recycle bin 脚本 |

**Track A 总计：19d（可压缩为 3 周，因 A1/A2 并行）**

### Track B — Go 后端线

| 序号 | 模块 | 工时 | 前置依赖 | 关键文件 |

| B1 | Token 算力配额管理 | 9d | 无 | `management-server/internal/token_quota/` (新增) |
| B2 | 影子智能体检测 | 4d | 无 | `management-server/internal/` + heartbeat 端 |
| B3 | 三级审批流 | 7d | 无 | OPA Rego + `management-server/internal/policy/` |
| B4 | 企业 IAM + ABAC JIT | 12d | 无（建议 Redis 完成后做） | OPA authz.rego + 新 JIT handler |

## 八、风险管理

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| AttackAgent 训练数据不足 | 中 | Track C 延期 | 先用 ADFA-LD + 合成数据，后期补充真实攻击样本 |
| FAISS 与现有 150D 向量格式对齐问题 | 低 | Track C 延期 | FAISS 支持 float32 标准格式，CAE 输出可直接对接 |
| SELinux 策略与 Docker privileged 模式冲突 | 中 | Track A 延期 | 先在 VM 裸机测试，再适配容器环境 |
| IAM 协议适配复杂（LDAP/SAML/OIDC） | 高 | Track B 延期 | v1 仅支持 OIDC，其余协议按需扩展 |
| 欺骗式防御 Mock 规则与 OPA 策略不一致 | 中 | Track B 延期 | Mock 规则从 OPA 策略自动派生，避免手动维护 |
